# Explore Nanobot and WebUI

**Time:** 10:15-10:30

## Purpose
Create a quick first success with Nanobot as the shared agent system for the day.

## Key Question
What can Nanobot already do, and what feels immediately useful or surprising?

## Objectives
- Open Nanobot WebUI and test basic agent interaction.
- Identify useful capabilities and visible limits of the system.
- Surface first impressions from technical, research, PR, Living Lab, and non-coding perspectives.
- Build a shared intuition for what agentic assistants can do beyond this specific workshop.

## Context
- Nanobot is a lightweight agent system with WebUI, tools, workspace interaction, model routing, memory/session features, and deployment options.
- It can be understood as part of the broader family of OpenClaw-like agent systems, but the block does not require OpenClaw background knowledge.
- The point is to experience what an agent can do, where it feels useful, and where boundaries become visible.

## Exploration Tasks

Pick a few short tasks. Do not try to exhaust the system; the goal is shared first impressions.

### General
- Ask Nanobot to explain what kind of assistant it is and what it can help with.
- Ask for a short plan for a realistic everyday task, such as organizing notes, drafting an email, or preparing a small checklist.
- Ask it to explain what tools or capabilities it can use in this WebUI session.
- Ask where its answer may be uncertain or where human review is required.

### Non-Coders
- Ask Nanobot to rewrite a technical paragraph in plain language.
- Ask it to create a checklist for a small event, meeting, or documentation task.
- Ask it to explain a term you encounter in the interface without using developer jargon.
- Observe which answers are directly useful and which still need human judgment.

### PR and Communication
- Ask for three versions of the same explanation: visitor-friendly, press-friendly, and internal team note.
- Ask it to identify unclear wording in a short text.
- Ask it to propose accessibility-friendly labels or status messages for an interactive demo.
- Check whether the output is accurate, understandable, and suitable for public communication.

### Research
- Ask it to summarize a concept for a non-expert audience and then for a technical audience.
- Ask it to list assumptions, missing context, and possible hallucination risks in its own answer.
- Ask for questions that would help evaluate whether an AI demonstrator is scientifically honest.
- Observe where expert correction is still necessary.

### Developers and HPC/Operations
- Ask it to outline the steps needed to operate a small web-based demonstrator.
- Ask what logs, secrets, ports, and restart behavior should be checked before a public demo.
- Ask it to explain the difference between a quick workshop prototype and a maintainable deployment.
- Observe where the system gives concrete operational help and where it stays generic.

## Reflection Questions
- Which parts already feel like a demonstrator?
- Which parts feel more like an internal agent workbench than a public visitor interface?
- What would PR, accessibility, HPC, research, and development each worry about?
- What could help in daily work even outside coding?
- What should we not customize today because Nanobot already provides enough?

## Expected Output
- List of useful existing Nanobot capabilities.
- List of visible limits, risks, or unclear areas.
- First role-specific observations from non-coders, PR, research, development, and operations.

## Moderation Notes
- Start from the prepared RPi WebUI.
- Keep the block short and concrete; choose representative prompts rather than many prompts.
- Avoid chat-app integrations and messenger tokens.
- Do not imply that Nanobot already knows the retreat context unless that context has been explicitly provided.
- Capture observations in plain language so they can feed the next comparison block.
