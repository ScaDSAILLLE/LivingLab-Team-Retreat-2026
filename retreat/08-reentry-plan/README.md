# Re-entry and Afternoon Plan

**Time:** 14:00-14:10

## Purpose
Re-establish focus after lunch, introduce the feature tracks, and define realistic afternoon outputs.

## Key Question
Where are we now, what comes next, and what can each feature track realistically demonstrate within the available time?

## Objectives
- Review the shared baseline.
- Explain why the afternoon uses feature tracks: Scaddy is more than a chatbot and should support Living Lab visitors in several useful ways.
- Invite participants to ask what would be important, interesting, and feasible to implement or assess today.
- Assign participants to tracks with mixed perspectives; participants can contribute their own ideas too.
- Define one visible success criterion per track.

## Track Framing
Feature tracks are short experiments, not full product work. Each track should answer one practical question: what would make Scaddy more useful, understandable, or reliable for Living Lab visitors?

Use the planned tracks as starting points:
- Knowledge and Explanation: what should Scaddy know and how should it explain things?
- Eyes for Scaddy: how could images or exhibit photos become useful input?
- Voice for Scaddy: how realistic are speech input and output on this foundation?
- Demo Hardening: what makes the prototype stable, clear, accessible, and presentation-ready?

Participants may suggest another track if it has a clear public-demo value and a realistic output for the afternoon.

## Selection Questions
- Which track needs your perspective most?
- What can be demonstrated, tested, or clearly assessed in 30-45 minutes?
- What can be documented even if implementation fails?
- What would be useful for Living Lab visitors, PR, operations, research, or future development?

## Expected Output
- Track assignments.
- One visible success criterion per track.
- Known dependencies before the deployment block.

## Moderation Notes
- Avoid letting all developers cluster in one track.
- Ensure each track has at least one user, communication, testing, or documentation perspective.
- Make explicit that a clear integration assessment can be a valid result if implementation is too large.
- Keep this block short; detailed work starts in Block 09.
