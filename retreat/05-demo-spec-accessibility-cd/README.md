# Demo Design, Accessibility, Corporate Design, and UI Spec

**Time:** 11:30-12:30

## Purpose
Define the public-demo specification based on Nanobot WebUI, accessibility requirements, corporate design, and the Scaddy-V2 demonstrator goal.

## Key Question
What makes the demonstrator understandable, accessible, visually consistent, and suitable for public use?

## Objectives
- Decide what must change in or around Nanobot WebUI for a Living Lab demonstration.
- Capture accessibility, language, corporate design, and interaction requirements before feature work.
- Produce a shared `coding-spec.md` or equivalent spec that guides afternoon work.

## Suggested Time Split
- 10 minutes: PR/communications input on accessibility, clarity, and public wording.
- 10 minutes: Living Lab input on templates, visual language, visitor context, and Scaddy reference.
- 20 minutes: group discussion and tradeoff decisions.
- 20 minutes: write the shared specification.

## Spec Topics
- Target audience and demo setting.
- Supported languages.
- Tone of voice and explanation depth.
- Status messages and fallback wording.
- Accessibility requirements.
- Corporate design constraints.
- What to keep from Nanobot WebUI.
- What to wrap, configure, simplify, or postpone.
- What must be ready for a public pitch at the end of the day.

## Discussion Questions
- What would confuse a first-time visitor?
- What should be visible when the system is listening, thinking, answering, or failing?
- Which visual elements communicate trust and accessibility?
- Where do PR clarity and technical accuracy need negotiation?
- Which UI changes are actually necessary today, and which are aesthetic rabbit holes?

## Expected Output
- Shared demo/coding spec.
- Decision whether to use Nanobot WebUI as-is, configure it, reverse-proxy it, or wrap it later.
- Clear acceptance criteria for afternoon feature tracks.
